## RÈGLES ABSOLUES

- Zéro redondance. Si une logique se répète, elle est extraite dans une fonction ou un package partagé.
- Chaque endpoint valide ses inputs. Jamais de données non validées qui atteignent la base.
- Gestion d'erreur explicite partout. Jamais d'erreur silencieuse.
- Toutes les variables d'environnement passent par un fichier de config centralisé chargé au démarrage.
  Jamais d'accès direct à `os.Getenv` dispersé dans le code.
- Les réponses JSON sont toujours structurées :
  Succès : `{ "success": true, "data": { ... } }`
  Erreur  : `{ "success": false, "error": "message en français" }`
- Les codes HTTP doivent être sémantiquement corrects :
  200 succès, 201 création, 400 données invalides, 401 non authentifié,
  403 non autorisé, 404 introuvable, 409 conflit, 500 erreur serveur
- Zéro logique métier dans les handlers. Les handlers appellent des services.
  Les services appellent des repositories. Les repositories parlent à la base.
- Tests automatisés inclus (voir section dédiée).

## DOMAINE MÉTIER

### Rôles

4 rôles : `admin`, `teacher`, `student`, `company`.
L'admin hérite de toutes les fonctionnalités du teacher.
Il est unique et peut transférer son rôle à un enseignant.

### Entités principales

**Profile** : id, role, full_name, email, avatar_url, is_active

**Teacher** : lié à Profile — grade (assistant/mab/maa/mcb/mca/professeur),
department, availability_status (disponible/indisponible/indisponible_jusqu_au),
unavailable_until (date), domaines (N-N)

**Student** : lié à Profile — student_number, specialty_id, level, promotion_id

**Company** : lié à Profile — company_name, sector, description, logo_url,
contact_email, contact_phone, website, is_verified

**Speciality** : id, name, code (ISIL/CHIM/ELEC), year_type (licence/master)

**Domain** : id, name
Relations N-N : teacher_domains, subject_domains

**Promotion** : id, label, academic_year

**AcademicYear** : id, label, status (active/cloturee), submission_open_at,
submission_close_at, max_wishes

**PfeSubject** : id, title, description, group_type (monome/binome/trinome),
proposer_id, proposer_role, company_id, academic_year_id,
validator1_id, validator2_id, validator1_decision, validator2_decision,
validator1_comment, validator2_comment,
status (en_attente/valide/accepte_sous_reserve/refuse/expire),
co_supervisor_id, domaines (N-N), pre_assigned_student_ids

**Wish** : id, student_id, subject_id, academic_year_id,
status (en_attente/accepte/refuse)

**PfeAssignment** : id, pfe_code, subject_id, academic_year_id,
student_id, student2_id, student3_id,
supervisor_id, co_supervisor_id,
memoire_url, status (en_cours/memoire_soumis/soutenance_planifiee/valide/refuse)

**PfeProgressReport** (meeting) : id, assignment_id, meeting_date, duration,
meeting_type (presentiel/visio), topics, status (en_cours/termine), observation

**DefenseJury** : id, assignment_id, president_id, member_id,
president_confirmed, member_confirmed,
president_wants_printed, member_wants_printed

**Defense** : id, assignment_id, jury_id, scheduled_at, room,
defense_deadline, status (scheduled/done/postponed),
result (admitted/corrections_required/not_admitted), final_grade

**JuryGrade** : id, defense_id, jury_member_id,
criterion1..4 (float, /4 chacun, total /16)

**SupervisorEvaluation** : id, pfe_assignment_id (unique), evaluator_id,
criterion5 (float, /4)

Note finale = criterion1+2+3+4 (jury) + criterion5 (encadrant) = /20

**CompanyReport** : id, company_id, submitted_by, correction_type,
description, requested_value, status (en_attente/resolu/rejete), resolved_at

**Notification** : id, recipient_id, type, payload (JSON), read_at

**AuditLog** : id, actor_id, action, entity, entity_id, metadata (JSON)

### Génération du code PFE

Au moment de l'assignation des étudiants :
format : `PFE-[CODE_SPECIALITE]-[ANNEE]-[N paddé 3 chiffres]`
exemple : `PFE-ISIL-2025-001`

### Système de notation

5 critères × 4 points = 20 points total.
Critères 1-4 : soumis par chaque membre du jury indépendamment.
Critère 5 : soumis par l'encadrant du PFE.

Workflow :
1. Chaque jury soumet ses criterion1..4
2. Le président vérifie que le membre a soumis
3. Si oui, le président choisit : sa note | note du membre | nouvelle évaluation
4. Note finale = 4 critères jury choisis + criterion5 encadrant
